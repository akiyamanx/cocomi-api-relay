## 1. 機能一覧

### 1-1. 基本機能
- モノレポ内で relay Worker と agent-hub Worker を分離
- agent-hub Worker の新設
- 同一D1共有
- migration管理

### 1-2. タスク管理
- タスク作成
- タスク取得
- タスク一覧取得
- 承認
- 却下
- キャンセル
- 実行状態管理
- リトライ管理
- 親子タスク/チェーンタスク管理

### 1-3. 実行制御
- executor によるタスク実行
- step単位の実行記録
- stale task 回収
- timeout管理
- retry制御
- scheduled task 実行
- recurring task 基盤

### 1-4. コスト制御
- 日次コスト集計
- タスク単位コスト集計
- 警告閾値チェック（80%でソフト警告）
- 上限到達時停止（日次$1.00ハード停止、月次$10.00ハード停止）
- 監査ログ連携
- 通知連携

### 1-5. 権限制御
- resource/action 単位の権限判定
- 緊急停止時の実行拒否
- maintenance mode 時の read-only 化
- 日次呼び出し回数制御
- 承認要否フラグ制御

### 1-6. 監査
- task作成、承認、開始、完了、失敗の記録
- permission denied の記録
- cost limit hit の記録
- emergency stop の記録
- system error の記録

### 1-7. ヘルス/状態確認
- `GET /agent/health` — 基本ヘルスチェック
- `GET /agent/status` — 稼働中タスク件数・日次/月次コスト・config参照・異常フラグ

### 1-8. 通知
- 要承認タスク通知
- コスト80%通知
- コスト上限通知
- task failed 通知
- emergency stop 通知
- タイムアウト時の事後報告通知
- UX方針: 冷たいシステム文ではなく、理由・現在状態・次の選択肢を含め「家族の相談」の温度感で届ける

## 2. 画面構成・UI設計

### 2-1. 想定UI
- LINE Flex Message（メイン通知チャネル）
- 状態確認API（LINEキーワード「状況教えて」→ relay経由でagentに問い合わせ）

### 2-2. 通知UI方針
- 承認待ちタスクは自然文で提案する
- 生のJSONや内部エラーをそのまま見せない
- エラーは感情翻訳した文面に変換する
- 自動停止時は「弱音」表現を使い、相談/報告の温度感を維持する
<!-- ここちゃん提案: 「ごめんね、予算超えちゃいそうだから一旦ブレーキ踏んでるよ💦」的な温度感 -->

### 2-3. 承認フローUI要件（LINE Flex Message）
- タスクタイトル（40字以内）
- 説明/要約（60字以内）
- 想定コスト表示
- リスクレベル色分け（low=緑, medium=黄, high=赤）
- 承認ボタン「✅ いいよ！お願い」（Postback）
- 詳細ボタン「💬 詳細を教えて」（Postback）
- 却下ボタン「❌ 今回はやめとこ」（Postback）

### 2-4. 状態確認レスポンス要件
- running 件数
- pending approval 件数
- stopped / failed 件数
- 今日のコスト
- 今月のコスト
- 最終イベント時刻
- 異常フラグ（emergency_stop / maintenance_mode）

## 3. データ構造

### 3-1. テーブル一覧
1. `users` — ユーザー管理
2. `permissions` — 権限定義
3. `proposals` — 提案/タスク本体
4. `proposal_actions` — 状態遷移ログ
5. `cost_log` — コスト記録
6. `audit_log` — 監査ログ

### 3-1-1. タスク状態遷移図
```
[draft] → [submitted] → (承認要) → [approved] → [scheduled] → [running] → [completed]
                ↓                        ↓                          ↓
         [revision_requested]       [rejected]                   [failed] → (retry可) → [running]
                ↓                                                   ↓
            [submitted]                                        [cancelled]
                                                                    ↑
                                                            (どの状態からでも遷移可)

[running] → [paused] → [running]  (一時停止・再開)
[running] → [stopped]             (手動停止 or system_auto_stop)

※ emergency_stop 発動時: running → stopped（全件）
※ stale回収: running（timeout超過） → failed or running（retry）
```

### 3-2. CREATE TABLE SQL（0002_agent_hub.sql）

```sql
-- 0002_agent_hub.sql: Phase 1 初期テーブル

-- 1. users
CREATE TABLE IF NOT EXISTS users (
  id TEXT PRIMARY KEY,
  line_user_id TEXT UNIQUE,
  display_name TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('owner', 'admin', 'user', 'viewer')),
  status TEXT NOT NULL CHECK (status IN ('active', 'inactive')),
  timezone TEXT NOT NULL DEFAULT 'Asia/Tokyo',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

-- systemユーザー初期投入
INSERT OR IGNORE INTO users (id, line_user_id, display_name, role, status, created_at, updated_at)
VALUES ('system', NULL, 'System', 'admin', 'active', datetime('now'), datetime('now'));

-- アキヤ初期投入（line_user_idは後で更新）
INSERT OR IGNORE INTO users (id, line_user_id, display_name, role, status, created_at, updated_at)
VALUES ('akiya', NULL, 'アキヤ', 'owner', 'active', datetime('now'), datetime('now'));

-- 2. permissions
CREATE TABLE IF NOT EXISTS permissions (
  id TEXT PRIMARY KEY,
  subject_type TEXT NOT NULL CHECK (subject_type IN ('user', 'role')),
  subject_id TEXT NOT NULL,
  resource TEXT NOT NULL,
  action TEXT NOT NULL,
  effect TEXT NOT NULL CHECK (effect IN ('allow', 'deny')),
  conditions_json TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

-- 初期権限: ownerは全操作許可
INSERT OR IGNORE INTO permissions (id, subject_type, subject_id, resource, action, effect, created_at, updated_at)
VALUES ('perm_owner_all', 'role', 'owner', '*', '*', 'allow', datetime('now'), datetime('now'));

-- 初期権限: systemはstop/auto_stopのみ
INSERT OR IGNORE INTO permissions (id, subject_type, subject_id, resource, action, effect, created_at, updated_at)
VALUES ('perm_system_stop', 'user', 'system', 'proposal', 'stop', 'allow', datetime('now'), datetime('now'));

INSERT OR IGNORE INTO permissions (id, subject_type, subject_id, resource, action, effect, created_at, updated_at)
VALUES ('perm_system_auto_stop', 'user', 'system', 'proposal', 'system_auto_stop', 'allow', datetime('now'), datetime('now'));

-- 3. proposals
CREATE TABLE IF NOT EXISTS proposals (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  description TEXT,
  proposer_user_id TEXT NOT NULL REFERENCES users(id),
  mode TEXT NOT NULL CHECK (mode IN ('auto', 'assist', 'manual')),
  status TEXT NOT NULL CHECK (status IN (
    'draft', 'submitted', 'revision_requested', 'approved', 'rejected',
    'scheduled', 'running', 'paused', 'completed', 'failed', 'cancelled', 'stopped'
  )),
  risk_level TEXT NOT NULL CHECK (risk_level IN ('low', 'medium', 'high')),
  estimated_cost_micro_usd INTEGER NOT NULL DEFAULT 0,
  approved_budget_micro_usd INTEGER NOT NULL DEFAULT 0,
  actual_cost_micro_usd INTEGER NOT NULL DEFAULT 0,
  auto_stop_enabled INTEGER NOT NULL DEFAULT 1,
  execution_deadline_at TEXT,
  submitted_at TEXT,
  approved_at TEXT,
  started_at TEXT,
  completed_at TEXT,
  stopped_at TEXT,
  stop_reason TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

-- 4. proposal_actions
CREATE TABLE IF NOT EXISTS proposal_actions (
  id TEXT PRIMARY KEY,
  proposal_id TEXT NOT NULL REFERENCES proposals(id),
  action TEXT NOT NULL CHECK (action IN (
    'create', 'submit', 'request_revision', 'revise', 'approve', 'reject',
    'schedule', 'start', 'pause', 'resume', 'complete', 'fail', 'cancel', 'stop', 'system_auto_stop'
  )),
  actor_user_id TEXT NOT NULL REFERENCES users(id),
  from_status TEXT NOT NULL,
  to_status TEXT NOT NULL,
  comment TEXT,
  metadata_json TEXT,
  created_at TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_proposal_actions_proposal_id ON proposal_actions(proposal_id);
CREATE INDEX IF NOT EXISTS idx_proposal_actions_created_at ON proposal_actions(created_at);

-- 5. cost_log
CREATE TABLE IF NOT EXISTS cost_log (
  id TEXT PRIMARY KEY,
  proposal_id TEXT REFERENCES proposals(id),
  action_type TEXT NOT NULL,
  provider TEXT NOT NULL,
  model TEXT,
  input_tokens INTEGER DEFAULT 0,
  output_tokens INTEGER DEFAULT 0,
  cost_micro_usd INTEGER NOT NULL DEFAULT 0,
  daily_key TEXT NOT NULL,
  monthly_key TEXT NOT NULL,
  metadata_json TEXT,
  created_at TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_cost_log_daily_key ON cost_log(daily_key);
CREATE INDEX IF NOT EXISTS idx_cost_log_monthly_key ON cost_log(monthly_key);
CREATE INDEX IF NOT EXISTS idx_cost_log_proposal_id ON cost_log(proposal_id);

-- 6. audit_log
CREATE TABLE IF NOT EXISTS audit_log (
  id TEXT PRIMARY KEY,
  actor_user_id TEXT NOT NULL,
  action TEXT NOT NULL,
  resource_type TEXT NOT NULL,
  resource_id TEXT,
  detail_json TEXT,
  ip_address TEXT,
  created_at TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_audit_log_actor ON audit_log(actor_user_id);
CREATE INDEX IF NOT EXISTS idx_audit_log_created_at ON audit_log(created_at);
CREATE INDEX IF NOT EXISTS idx_audit_log_resource ON audit_log(resource_type, resource_id);
```

## 5. コスト制御設計

### 5-1. 3層コスト制御（構想カプセルv0.3準拠）

| レイヤー | 閾値 | 動作 |
|---------|------|------|
| タスク単位 | 軽調査$0.10 / 深掘り$0.50 / 会議$1.00 | 超過時に縮退 |
| 日次 | ソフト$0.70 / ハード$1.00 | ソフト=通知、ハード=全停止 |
| 月次 | ハード$10.00 | 全停止 |

### 5-2. コスト計算方式
- 各API呼び出し後にレスポンスのusageフィールドからトークン数×単価で計算
- `cost_log`テーブルにmicro_usd単位（$1.00 = 1,000,000 micro_usd）で記録
- `daily_key`は`YYYYMMDD`形式、`monthly_key`は`YYYYMM`形式

### 5-3. コスト設定（wrangler.agent.toml）
```toml
[vars]
DAILY_COST_HARD_LIMIT_MICRO_USD = "1000000"   # $1.00
MONTHLY_COST_HARD_LIMIT_MICRO_USD = "10000000" # $10.00
DAILY_COST_SOFT_LIMIT_MICRO_USD = "700000"     # $0.70
```

## 6. 安全設計

### 6-1. D1安全ガード
```javascript
// shared/d1-helpers.js
const FORBIDDEN_SQL_PATTERNS = [
  /DROP\s+TABLE/i,
  /ALTER\s+TABLE/i,
  /TRUNCATE/i,
  /DELETE\s+FROM\s+(?!cost_log|audit_log|proposal_actions)/i,
];

function validateSQL(sql) {
  for (const pattern of FORBIDDEN_SQL_PATTERNS) {
    if (pattern.test(sql)) {
      throw new Error(`[SECURITY] Forbidden SQL pattern: ${sql.substring(0, 50)}`);
    }
  }
}
```

### 6-2. 保護対象テーブル
agent-hubから書き込み禁止のテーブル:
- `memories`
- `memory_metadata`
- relay側が管理するすべてのテーブル

### 6-3. 緊急停止 / メンテナンスモード
- `emergency_stop`: 全書き込み系APIの入口でチェック。trueなら即拒否
- `maintenance_mode`: read-only化。GETのみ許可

## 7. リスク分析と対策

| リスク | 重大度 | 対策 |
|--------|--------|------|
| agent側バグがrelay側テーブルを破壊 | 高 | 破壊的SQL禁止パターン + CLAUDE.mdに明文化 |
| agentデプロイ失敗でrelayも巻き添え | 中 | 別Worker→デプロイ完全独立 |
| D1の同時書き込み競合 | 低 | 異なるテーブルに書くため衝突しない |
| 認証トークン漏洩 | 高 | wrangler secretで管理。ログに出さない |
| コスト自動停止が効かない | 高 | WRITE前に必ずチェック。チェック失敗時もWRITE停止 |
| LINE Flex Messageペイロード上限 | 低 | タイトル40字・要約60字でtruncate |
| runningタスクの取りっぱぐれ | 中 | scheduler.jsで300秒超過をstale判定→failed or retry |

## 8. 既存Workerのモノレポ移行手順（ゼロダウンタイム）

```
Step 1: ディレクトリ構造作成
  mkdir -p workers/relay workers/agent-hub migrations/relay migrations/agent shared

Step 2: 既存ファイルをworkers/relay/に移動
  git mv src/index.js workers/relay/
  git mv src/memory.js workers/relay/
  git mv src/import.js workers/relay/
  git mv src/vector.js workers/relay/
  git mv src/search.js workers/relay/
  git mv src/utils.js workers/relay/

Step 3: workers/relay/wrangler.tomlを作成（既存wrangler.tomlの内容コピー）
  ※ name = "cocomi-api-relay" は維持

Step 4: ルートのwrangler.tomlを削除

Step 5: workers/relay/ からデプロイして動作確認
  cd workers/relay && wrangler deploy
  ※ ここで失敗したら元に戻すだけ

Step 6: agent-hub のファイルを追加（まだデプロイしない）

Step 7: マイグレーション実行
  wrangler d1 execute cocomi-db --file=./migrations/agent/0002_agent_hub.sql

Step 8: agent-hub をデプロイ
  cd workers/agent-hub && wrangler deploy

Step 9: relay側にagentへのルーティング追加
  cd workers/relay && wrangler deploy
```

**重要: Step 5の動作確認が最優先。** Termuxでwranglerが動かない場合はCloudflareダッシュボードから手動デプロイ or GitHub Actions経由で対応。
